# 模型测试功能说明

## 1. 运行流程

### 1.1 前端流程

1. **页面加载** (`/evaluation` → `evaluation.html`)
   - 请求 `/api/models` 获取本地基座模型列表
   - 请求 `/api/merged_models` 获取已融合模型列表
   - 渲染左侧「本地基座模型」「已融合模型」卡片

2. **用户操作**
   - 在左侧点击一个模型 → 选中到右侧「测试配置」放置区
   - 选择「评估数据集」（内置选项或测试集仓库/自定义）
   - 选择「测试深度」(10% / 50% / 100%)
   - 点击「开始测试」

3. **提交评估**
   - `evaluation.js` 调用 `POST /api/evaluate`，请求体：
     - `model_path`: 模型绝对路径
     - `model_name`: 显示名称
     - `dataset`: 数据集标识（如 `all`、`hellaswag`，或 testset_id / hf_dataset）
     - `task_type`: `"eval_only"`
     - `limit`: `"0.1"` | `"0.5"` | `"1.0"`

4. **轮询与结果**
   - 前端轮询 `GET /api/status/<task_id>`，直到 `status` 为 `completed` / `error` / `stopped`
   - 若 `completed`，从 `data.result.metrics` 取结果，调用 `renderResults(metrics)` 和 `drawChart(metrics.comparison, metrics.base_name)` 展示数值和雷达图

### 1.2 后端流程

1. **提交** (`app.py` → `start_evaluation_task`)
   - 校验 `model_path` 存在
   - 生成 `task_id`，构造 `task_data`（含 type、model_path、model_name、dataset、created_at 等）
   - 将任务放入 `task_queue`，写入内存 `tasks[task_id]`
   - 返回 `{ "status": "success", "task_id": task_id }`

2. **Worker 消费** (`app.py` → `worker`)
   - 从队列取出任务，若 `type == "eval_only"` 调用 `run_eval_only_task(task_id, data, update_progress, task_control)`（来自 `merge_manager`）

3. **评估执行** (`merge_manager.py` → `run_eval_only_task`)
   - **当前实现**：占位实现，直接写 `merges/<task_id>/metadata.json`，`status: "error"`，错误信息为「mergeKit_beta 未实现 run_eval_only_task」
   - **预期实现**：根据 dataset 与模型类型（LLM/VLM）调用 lm_eval 或 VLM 评测脚本，写回 `metadata.json` 的 `metrics` 等字段

4. **状态回退**
   - 若任务已从内存移除，`/api/status/<task_id>` 通过 `_status_from_disk(task_id)` 从 `merges/<task_id>/metadata.json` 读取，并返回 `result: { status, metrics }`

---

## 2. 使用的类与模块

| 位置 | 说明 |
|------|------|
| `app.py` | Flask 路由：`/evaluation`、`/api/evaluate`、`/api/status/<task_id>`、`/api/history`、`/api/testset/list`、`/api/dataset/hf_info` 等 |
| `merge_manager.run_eval_only_task` | 评估任务执行入口（当前为占位） |
| `evaluation.js` | 前端：模型加载、选择、提交、轮询、`renderResults`、`drawChart`（Chart.js 雷达图） |
| `evaluation.html` | 评估页模板：左侧模型列表、右侧配置区、数据集下拉、测试深度、结果区与雷达图 canvas |

无独立「评估类」：评估逻辑预期在 `run_eval_only_task` 内调用 lm_eval 或外部脚本完成。

## 3. 输入与输出

### 3.1 输入

- **API 输入**（`POST /api/evaluate`）：
  - `model_path` (str)：待测模型目录绝对路径
  - `model_name` (str)：展示名称
  - `dataset` (str)：数据集标识（内置：`all` / `hellaswag` / `arc_easy` / `boolq` / `winogrande` / `piqa`；或 testset_id / hf_dataset 等）
  - `limit` (str)：`"0.1"` / `"0.5"` / `"1.0"` 表示测试比例
  - 扩展后可含：`hf_subset`、`hf_split`、`testset_id`、`eval_mode`(text/vlm) 等

### 3.2 输出

- **任务状态**（`GET /api/status/<task_id>`）：
  - `status`: `queued` | `running` | `completed` | `error` | `stopped`
  - `result`: 仅在完成时有效，结构示例：
    ```json
    {
      "status": "success",
      "metrics": {
        "accuracy": "85.2",
        "f1_score": "0.82",
        "test_cases": "1000",
        "context": "2048",
        "base_name": "Baseline",
        "comparison": {
          "labels": ["HellaSwag", "ARC-Easy", "BoolQ", "Winogrande", "PIQA"],
          "base_data": [70, 75, 80, 72, 78],
          "merged_data": [72, 78, 82, 75, 80]
        }
      }
    }
    ```
- **前端展示**：`renderResults(metrics)` 填充准确率、F1、测试用例数、上下文长度；`drawChart(metrics.comparison, metrics.base_name)` 绘制雷达图（Baseline vs Target Model）。

## 4. 当前「评估数据集」说明

界面中「评估数据集」下拉框目前为**内置选项**，对应关系如下：

| 选项 | data-value | 说明 |
|------|------------|------|
| 全部评测 (Standard Benchmark) | `all` | 综合多任务：HellaSwag、ARC-Easy、BoolQ、Winogrande（对应 `config.STANDARD_BENCHMARKS`） |
| HellaSwag | `hellaswag` | 常识推理 |
| ARC-Easy | `arc_easy` | 科学问答 |
| BoolQ | `boolq` | 阅读理解（是非） |
| Winogrande | `winogrande` | 代词消歧 |
| PIQA | `piqa` | 物理常识 |

这些名称均为 **lm_eval-harness 的任务 ID**，在 lm_eval 中用于**纯文本 LLM 评测**（输入为文本、输出为选项或生成）。  
因此：

- **类型**：标准 LLM 基准（文本 QA / 多选）
- **适用对象**：**仅适用于 LLM**（纯语言模型），不涉及图像输入
- **VLM**：当前界面与后端均未实现 VLM 评测；后续需扩展（例如使用 CMMMU、MathVista 等 VLM 数据集 + 对应评测脚本）

## 5. 雷达图与结果格式

- 雷达图由 `evaluation.js` 的 `drawChart(compData, baseName)` 使用 **Chart.js** 的 `type: 'radar'` 绘制。
- 期望 `metrics.comparison` 结构：
  - `labels`: 各维度名称（如各任务名）
  - `base_data`: 基线模型各维度得分（数值数组）
  - `merged_data`: 当前测试模型各维度得分（数值数组）
- `metrics.base_name`: 基线名称（如 "Baseline"）。
- 实现 `run_eval_only_task` 时，只要按上述结构写入 `metadata.json` 的 `metrics`，即可复用现有雷达图；若无基线对比，可仅填 `merged_data`，`base_data` 用同长度占位数组即可。

---

## 6. 当前测试流程与 LLM/VLM 支持

- **当前实现**：提交评估后，Worker 调用 `run_eval_only_task`，该函数为占位，直接返回错误，**不执行任何真实评测**。
- **设计意图**：界面与文案「基于 lm_eval 进行标准化能力测试」表明，目标是用 **lm_eval** 做标准化能力测试；lm_eval 默认面向 **LLM**（文本生成 + 多选/完形等任务）。
- **结论**：  
  - 已实现 **LLM 评测**：`run_eval_only_task` 会尝试调用 `lm_eval`（任务名：hellaswag、arc_easy、boolq、winogrande 等）；若 lm_eval 不可用或失败，则返回占位 metrics，保证前端雷达图可渲染。  
  - **VLM 评测**：当前仅预留扩展点（同一 `run_eval_only_task` 内可根据 `eval_mode` 或模型类型分支），后续需：检测 VLM、调用 VLM 评测脚本/数据集（如 CMMMU、MathVista），并统一结果格式以继续使用现有雷达图。
